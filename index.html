JWT (JSON Web Tokens)

Today I will tell you about JSON Web Tokens. JWT is the way we can manage users' access in our applications. But before we start with JWT itself, let's find out the differences between two things: Authentication & Authorization.

Authentication & Authorization

Authentication and authorization might sound similar, but they are distinct security processes. Authentication confirms that users are who they say they are. Authorization gives those users permission to access a resource. Well, let's talk about what is authentication indeed. Authentication is the process of identifying users and validating who they claim to be. One of the most common and obvious factors to authenticate identity is a password. If the user name matches the password credential, it means the identity is valid, and the system grants access to the user. Authorization, on the other hand, happens after a userâ€™s identity has been successfully authenticated. It is about offering full or partial access rights to resources to get the job done. It means every user should have the right to access some information depending on the role.

What is JSON Web Token?

JSON web token (JWT), pronounced "jot", is an open standard (RFC 7519) that defines a way for securely transmitting information between parties as a JSON object.

JSON web token (JWT), pronounced "jot", is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties as a JSON object. It's important to note, JWT is a standard, meaning that all JWTs are tokens, but not all tokens are JWTs.

What is the structure of JWT?

JSON Web Token consists of 3 parts seprated by dots:

  • Header
  • Payload
  • Signature

Header

The header typically consists of two parts: the type of the token, which is JWT, and the signing algorithm.

              
                {
                  "alg": "HS256",
                  "typ": "JWT"
                }

Payload

The second part of the token is the payload, which contains the claims that represent additional data.

              
                {
                  "sub": "1234567890",
                  "name": "John Doe",
                  "iat": 1457454340,
                  "exp": 1457450740,
                  "admin": false
                }

!Notice that this information is readable by anyone. Do not put secret information in the payload or header elements of a JWT unless it is encrypted

The second part of the token is the payload, which contains the claims. Claims are statements about an entity (typically, the user) and additional data. "iat" and "exp" fields represent the number of seconds since January 1, 1970 (UNIX Epoch) and stand for "Issued At" and "Expires In" respectively. The payload is then Base64Url encoded to form the second part of the JSON Web Token.

Signature

To create the signature part you have to take the encoded header, the encoded payload, a secret, the algorithm specified in the header, and sign that.

To create the signature part you have to take the encoded header, the encoded payload, a secret, the algorithm specified in the header, and sign that. The signature is used to verify the message wasn't changed along the way, and, in the case of tokens signed with a private key, it can also verify that the sender of the JWT is who it says it is.

Access Token

Access token is used to get access to the resource directly. They usually have short expiration time.

There are 2 types of tokens: access token and refresh token. Access tokens carry the necessary information to access a resource directly. In other words, when a client passes an access token to a server managing a resource, that server can use the information contained in the token to decide whether the client is authorized or not. Access tokens usually have an expiration date and are short-lived.

Refresh Token

Refresh token provides the opportunity to get a new access token. It is used to create new access tokens after old ones have expired, or getting access to a new resource for the first time.

Access token <=30 minutes, refresh token - up to several months.

Refresh tokens carry the information necessary to get a new access token. In other words, whenever an access token is required to access a specific resource, a client may use a refresh token to get a new access token issued by the authentication server. Common use cases include getting new access tokens after old ones have expired, or getting access to a new resource for the first time. Refresh tokens can also expire but are rather long-lived. For example, lifetime of an access token is usually <= 30 minutes, refresh token may live up to several months. Refresh tokens are usually subject to strict storage requirements to ensure they are not leaked. They can also be blacklisted by the authorization server.

How to store tokens?

  • Access tokens should be stored in application memory.
  • Refresh tokens should however be stored in cookies with an httpOnly flag that prevents client-side scripts from accessing the data it contains.

Here's an example how to set httpOnly cookie using Express framework in Node.js.

              
                res.cookie('refreshToken', refreshToken, { httpOnly: true });

Login process

  1. User logs in the application passing login/password & browser's fingerprint.
  2. Server verifies login/password.
  3. In case of success server creates session and stores it in the database.
  4. Creates access token.
  5. Server sends both access and refresh tokens to the client.
  6. Client saves tokens in appropriate way: access token - in app memory, refresh token - in cookies.

Refresh process

  1. Client checks whether lifetime of an access token expired or not.
  2. When lifetime is over client makes a request to refresh the token passing such data as browser's fingerprint and refresh token.
  3. Server gets the refresh session record with the help of refresh token.
  4. Server saves current refresh session to the variable and deletes it from the database table.
  5. Server checks current session for lifetime and fingerprint.
  6. If it fails throws an error (like this: TOKEN_EXPIRED/INVALID_REFRESH_SESSION)
  7. In case of success server creates new refresh-session and saves it to the database.
  8. Server creates access token.
  9. Server sends access and refresh tokens to the client.

Important to note: when refreshing access token both access and refresh token should be updated.

What happens when tokens got stolen?

  1. A hacker uses access token.
  2. After a period of time this token's lifetime expired.
  3. The hacker sends refresh token and fingerprint.
  4. Server cannot find hacker's fingerprint in refresh session and deletes this session from DB.
  5. Server redirects user (hacker in our case) to login page, but hacker has no login or password.
  6. In such situation real user will be also redirected, but there's no problem, because user knows both login and password.

JWT vs Cookie Sessions

In Cookie Sessions the server is responsible for the authentication and the client does not know what happens at the server side after sending a request.

Before the emergence of JSON Web Tokens, we predominantly used session-based authentication. In this type of authentication method, the server is responsible for the authentication and the client does not know what happens at the server side after sending a request. And this is probably the main problem of Cookie Sessions.

Advantages of JWT

  • There's no need to ask for information in DB on every request.
  • Native mobile support of JWT
  • Ability to work with microservices.

Disadvantages of JWT

  • Token can be easily decoded.
  • Token is pretty big and works slower than traditional session.
In case there is a large amount of users accessing our app at once, we will face some server performance issues. Nowadays apps should work fine both in browsers and smartphones. And it's more convenient to work with JWTs when you develop mobile apps. It's worth noting that in so widely used today microservice architecture using JWTs is the only way to manage access in every service independently. When using sessions we should always send request to authentication server which is impossible in microservices. The main disadvantage of using JSON Web Tokens is the fact that data passed in its header and payload can be easily decoded. This means you should never set any sensitive data, such as credentials, password, etc., in token. In addition, token contains a large amount of data and works slower in comparison with session.
